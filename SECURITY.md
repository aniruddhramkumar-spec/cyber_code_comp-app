# Security Documentation - ChartVault

## Overview

ChartVault implements enterprise-grade security measures to protect user data and prevent common web application vulnerabilities. This document outlines the security architecture and best practices.

## 1. Authentication

### Multi-Factor Authentication (MFA)

**Implementation:**
- TOTP (Time-based One-Time Password) using RFC 6238
- 6-digit codes generated by authenticator apps
- 30-second time window with ±30-second clock skew tolerance

**Benefits:**
- Prevents account takeover via stolen credentials
- Requires physical access to authenticator device
- Industry-standard implementation

**User Guide:**
1. Enable MFA in Security Settings
2. Scan QR code with authenticator app
3. Verify with 6-digit code
4. Save backup codes in secure location

### Password Management

**Requirements:**
- Minimum 12 characters
- Must include: uppercase, lowercase, digits, special characters
- Enforced at registration and change
- Never transmitted in plain text

**Hashing:**
- Algorithm: bcrypt with 12 salt rounds
- Cost factor: 12 (security over speed)
- Salting: Automatic per-password

**Account Lockout:**
- 5 consecutive failed attempts triggers lockout
- 15-minute lockout period
- Prevents brute force attacks

## 2. Data Protection

### Encryption

**Symmetric Encryption:**
- Algorithm: AES-256 (via Fernet)
- Key: 256-bit random key
- Key Storage: `.data/encryption.key` with 0600 permissions
- Implementation: Python cryptography library

**Encrypted Data:**
- User MFA secrets (optional)
- Graph data (optional future enhancement)
- Sensitive parameters

**Key Management:**
- Auto-generated on first run
- Stored securely with restricted permissions
- Should be backed up in secure location
- Never commit to version control

### Database Security

**SQLite Hardening:**
- Parameterized queries (prepared statements)
- Input validation on all inputs
- No dynamic SQL construction
- Type enforcement

**Database Location:**
```
.data/graphs.db
```
Permissions: `0600` (read/write owner only)

## 3. Input Validation

### Validation Strategy

**All user inputs validated:**
1. Type checking
2. Length restrictions
3. Character whitelist validation
4. Numeric range checking
5. HTML sanitization

### Specific Validations

```python
# Usernames
- Length: 3-20 characters
- Characters: alphanumeric, -, _
- No special characters

# Passwords
- See password requirements above
- Length: 12-128 characters

# Graph Names
- Length: max 100 characters
- Characters: alphanumeric, spaces, basic punctuation
- No HTML/script tags

# Axis Labels
- Length: max 50 characters
- Allow basic special characters
- HTML escaped

# Numeric Data
- Valid IEEE 754 floating point
- Range: -1e308 to 1e308
- No NaN or Infinity values
- Decimal precision preserved
```

## 4. SQL Injection Prevention

### Parameterized Queries

All database operations use prepared statements:

```python
# ✓ SECURE
cursor.execute(
    "SELECT * FROM users WHERE username = ?",
    (username,)
)

# ✗ INSECURE (NOT USED)
cursor.execute(f"SELECT * FROM users WHERE username = '{username}'")
```

### Query Protection

- No string interpolation in SQL
- All variables pass through parameter binding
- Database escaping automatically applied
- Multiple layers of validation

## 5. Access Control

### Ownership Verification

**Graph Access:**
```python
# Verify user ownership before load/delete
def get_graph(graph_id, user_id):
    cursor.execute(
        "SELECT * FROM graphs WHERE id = ? AND user_id = ?",
        (graph_id, user_id)
    )
```

**Permission Model:**
- Users can only view/edit/delete their own graphs
- No cross-user data access
- Admin features not yet implemented

### Session Management

**Session Tracking:**
- User ID stored in session state
- Timeout after 1 hour of inactivity
- Logout clears all session data
- MFA verification per session

## 9. Incident Response

### Potential Security Issues

**If you discover a security vulnerability:**

1. **Do not** post it publicly
2. Contact the development team immediately
3. Provide detailed reproduction steps
4. Allow time for patch development
5. Help verify the fix

### Breach Response

1. **Immediate**: Disable affected accounts
2. **Assessment**: Identify scope of breach
3. **Communication**: Notify affected users
4. **Remediation**: Apply fix and verify
5. **Follow-up**: Update security measures
